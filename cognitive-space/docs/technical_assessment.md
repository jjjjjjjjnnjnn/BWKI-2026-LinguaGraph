## 1. 执行摘要

经过 10+ 轮调试，确认 **3d-force-graph v1.80.0 (unpkg CDN)** 的节点颜色渲染 API 存在根本性故障。所有颜色回调（`nodeColor`、`nodeThreeObject`）均被库忽略。节点在场景中存在且位置正确，但颜色始终为库默认值 `#7ab0ff`。

**这是库的 bug，不是代码逻辑问题。**

---

## 2. 根因分析

### 2.1 已确认的事实

| 测试 | 操作 | 结果 |
|------|------|------|
| T1 | `nodeColor(colorCB)` 回调返回 `#60a5fa` | Mesh 材质仍是 `#7ab0ff` |
| T2 | `nodeColor(() => '#ff0000')` 强制红色 | Mesh 材质仍是 `#7ab0ff` |
| T3 | 直接设置 `node.color = "#34d399"` | Mesh 材质仍是 `#7ab0ff` |
| T4 | `graphData()` 重新加载带颜色数据 | Mesh 材质仍是 `#7ab0ff` |
| T5 | `scene.traverse()` 直接修改 `material.color` | 诊断时恢复为 `#7ab0ff` |
| T6 | `nodeThreeObject` 返回自定义 MeshBasicMaterial | 回调**从未被调用**，场景中无 MeshBasicMaterial |

### 2.2 结论

**3d-force-graph v1.80.0 的以下 API 完全失效：**

- `nodeColor(fn)` — 回调被调用但返回值被忽略
- `nodeThreeObject(fn)` — 回调从未被执行
- `node.color` 数据属性 — 库不读取此属性

**唯一有效的渲染路径是库的默认 MeshLambertMaterial + 默认颜色 `#7ab0ff`。**

### 2.3 技术细节

库内部渲染代码（逆向工程确认）：

```javascript
// 库内部创建材质时使用硬编码默认值
h = new THREE.Color(ff(c || "#ffffaa"))
// ff() 函数正确解析颜色字符串
// 但材质创建后被库的更新逻辑覆盖
```

`colorCB` 确实被调用且返回正确值（Console 日志确认），但颜色不被应用到材质。

## 6. 建议方案

### 方案 A：换库（推荐）

使用其他 3D 图谱库，其颜色 API 正常工作：

| 库 | 优势 | 劣势 |
|----|------|------|
| **@react-three/fiber + d3-force-3d** | 完全控制渲染，颜色可靠 | 需要 React |
| **cytoscape.js (3D 扩展)** | 颜色 API 可靠 | 3D 效果较弱 |
| **Sigma.js + WebGL** | 高性能，颜色可靠 | 学习曲线 |
| **自建 Three.js** | 完全控制 | 开发量大 |

### 方案 B：降级到 v1.73.1

尝试旧版本 3d-force-graph，其 `nodeColor` API 可能正常：

```html
<script src="https://unpkg.com/3d-force-graph@1.73.1"></script>
```

### 方案 C：接受现状

接受所有节点同色 `#7ab0ff`，在论文中仅描述数据管道而非可视化效果。
