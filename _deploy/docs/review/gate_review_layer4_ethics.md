## Issue 1: Google Forms for Data Collection Contradicts Own Policy (CRITICAL)

- **Risk**: Unlawful data processor relationship with Google for minor data
- **Severity**: CRITICAL
- **Evidence**:
  - `docs/ethics/gdpr_package.md:128`: "Data Collection: All participant responses are collected via Google Forms"
  - `docs/ethics/gdpr_package.md:128`: "No data is stored on commercial cloud services" -- **direct contradiction with the line immediately preceding it**
  - `docs/experiment-design.md:35`: "1. 发放问卷（Google Forms / 纸质）"
  - `docs/compose/plans/2026-06-16-cognitivespace-v5.md:1017`: confirms Google Forms
- **Impact**: Google Forms routes data through US-based servers with no mention of a Data Processing Agreement (DPA) as required by GDPR Art. 28. For a study targeting minors (13-18), this is disqualifying for many German school authorities (Schulbehörden) and potentially violates Art. 44 (international transfers). The ethics package simultaneously claims "no cloud storage" and proposes Google Forms -- a reviewer will flag this inconsistency immediately.
- **Fix cost**: Low
- **Fix suggestion**: (1) Replace Google Forms with a self-hosted solution (LimeSurvey on a German server, or a local paper-only collection). (2) If Google Forms must be used, sign a DPA with Google, restrict to EEA data regions, and remove the "no cloud" claim. (3) Document the data processor relationship explicitly.

## Issue 2: Textbook Corpus (data/textbook/) Has No License or Source Attribution (HIGH)

- **Risk**: Copyright infringement from unattributed textbook content
- **Severity**: HIGH
- **Evidence**:
  - `data/textbook/zh_freedom.txt`: No header, no source URL, no license. Content appears to be a state-aligned textbook description of "freedom" ("自由是社会主义核心价值观的重要内容").
  - `data/textbook/de_freedom.txt`: No header, no source, no license. Content appears to be a curated textbook-like passage on "Freiheit."
  - `data/textbook/en_freedom.txt`: Same pattern -- no source, no license, no attribution.
  - All 15 `data/textbook/*.txt` files: zero attribution across the entire directory.
- **Impact**: BWKI is a German competition that requires CC-BY-SA or similarly open-licensed submission materials. Textbook content reproduced without permission or attribution cannot be legally included in a competition submission. If these are AI-generated, they should be labeled as such; if they are copied from textbooks, they require publisher permission or fallback to fair-use-for-research (which is narrow in German copyright law, UrhG §60a-60g).
- **Fix cost**: Low
- **Fix suggestion**: (1) Add a header to every textbook file with source, license, and retrieval date. (2) If AI-generated, add a generation note (e.g., "Generated by LLM on 2026-06-17 as a curricular exemplar, not a verbatim textbook extract"). (3) If from real textbooks, remove and replace with properly licensed material, or document the UrhG §60c research exception with a justification note.

## Issue 5: Data Deletion Mechanism Is Documented But Not Implemented (MEDIUM)

- **Risk**: GDPR Art. 17 right to erasure cannot be fulfilled in practice
- **Severity**: MEDIUM
- **Evidence**:
  - `docs/ethics/gdpr_package.md:130-146`: Deletion policy is documented -- locate ID in mapping file, delete from DB, confirm within 30 days
  - `docs/ethics/gdpr_package.md:130`: "The mapping between real names and IDs is stored separately in a password-protected file" -- **this file does not exist in the repository**
  - No deletion script found anywhere in `scripts/` or `src/`
  - `scripts/db_init.py` has no deletion/stored procedures
  - Deleting from a SQLite database manually is possible but error-prone and undocumented for a non-technical researcher
- **Impact**: If a parent or participant requests data deletion under GDPR Art. 17, there is no mechanism to (a) look up the participant ID from their name, (b) cascade-delete across responses, extractions, graphs, and analysis tables, or (c) confirm deletion. The name-ID mapping file doesn't exist, so even reconstruction is impossible. This is a concrete compliance gap, though the policy documentation is present.
- **Fix cost**: Medium
- **Fix suggestion**: (1) Create the name-ID mapping file or document its planned location/format. (2) Write a `scripts/delete_participant.py` script that takes a participant ID, cascades deletion across `responses`, `extractions`, `graphs`, and `cross_language_analysis` tables, and logs the action. (3) Consider soft-delete with `deleted_at` timestamp instead of hard-delete for audit trail. (4) Add a unit test that verifies deletion works.

## Additional Observations (Not Findings, But Worth Noting)

1. **Name-ID mapping file does not exist.** The gdpr_package.md references "a password-protected file" but no such file has been created. This makes both pseudonymization (GDPR Art. 4(5)) and the deletion procedure (Art. 17) inoperable.

2. **Cross-language analysis table lacks source tracking.** `scripts/db_init.py` defines `cross_language_analysis` but it has no `source` or `origin` column to indicate whether an analysis row was computed from human data, simulation data, or both. This makes it impossible to surgically recompute or delete analysis results for one participant.

3. **Social media scraper exists in repo.** `experiments/social_media_scraper.py` collects data from unspecified social media platforms. If this data is used in any analysis, it raises additional GDPR consent concerns (no consent from social media posters).

4. **Old consent_form.md exists at root level.** `data/consent_form.md` is a previous version (mentions "CognitiveSpace" and "calculus students") that is less comprehensive than the current package. This orphaned file should be removed to avoid confusion.

5. **Gold standard dataset sensitivity.** `data/gold/gold_dataset.json` contains human-annotated student misconceptions. If these annotations are derived from identifiable student data, they may not be truly anonymized. Consider whether these gold labels were obtained with consent.
